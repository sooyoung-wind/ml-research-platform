# v0.2.0 — Research Dashboard & Knowledge Graph

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

## Goal

사용자가 연구 주제를 쿼리하면:
1. 관련 논문을 수집·분석 (5W1H + 장단점)
2. 지식그래프(Cypher/SQLite)로 관계 구축
3. 트렌드 리포트 자동 생성
4. **HTMX 웹 대시보드**에서 논문 탐색 → 선택 → DeepCode 구현 요청

**핵심 철학:** CLI는 백그라운드 파이프라인. **Web이 사용자 인터페이스.** 사용자가 웹에서 논문을 보고, 분석을 확인하고, 구현을 요청한다.

```
│              HTMX Web Dashboard                  │
│  FastAPI + Jinja2 + HTMX + Tailwind CSS          │
│                                                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ Paper    │ │ Graph    │ │ Trend Report     │  │
│  │ List/    │ │ Viz      │ │ + Gap Analysis   │  │
│  │ Detail   │ │ (pyvis)  │ │                  │  │
│  └────┬─────┘ └──────────┘ └──────────────────┘  │
│       │ "Implement with DeepCode"                 │
│       ▼                                           │
│  ┌──────────────────────────────────────────┐    │
│  │ DeepCode Runner                          │    │
│  │ → 사용자 승인된 논문만 코드 구현         │    │
│  └──────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
         │                           │
         ▼                           ▼
  ┌──────────────┐          ┌───────────────┐
  │ Ollama       │          │ graphqlite    │
  │ gemma4:31b   │          │ (SQLite+Cypher│
```

**Per-Topic 독립 그래프:**
```

```

## Tech Stack

| Component | Technology | Why |
|-----------|-----------|-----|
| Web Framework | FastAPI + Jinja2 | Python 네이티브, HTMX와 찰떡 |
| Frontend Interactivity | HTMX | JS 없이 동적 UI, 오버엔지니어링 방지 |
| CSS | Tailwind CSS CDN | 빌드 없이 클래스만으로 스타일링 |

| Code Gen | DeepCode (deepcode-hku) | 논문 → 코드 구현 |

## Phase 구성

### Phase 5: Paper Analysis Engine (5W1H + 장단점)

**목표:** 논문을 5W1H 프레임워크로 구조화 분석

**분석 항목:**
- **Who:** 연구자, 소속, 연구 그룹
- **What:** 연구 문제, 기여점(Contribution), 핵심 주장
- **When:** 발행 시점, 관련 타임라인
- **Where:** 학회/저널, 연구 기관
- **Why:** 연구 동기, 기존 한계(Limitations), 해결 Gap
- **How:** 방법론, 데이터셋, 아키텍처, 실험 설정
- **Strengths:** 장점 (SOTA, 효율성, 일반성 등)
- **Weaknesses:** 단점 (제약, 비용, 한계)
- **Future Work:** 개선 방향 (저자 제시 + 분석 도출)
- **Reference Chain:** 인용한 선행 연구 목록
- **Evidence Sentences:** 각 추출 항목의 근거 문장 (Red Team 피드백 반영)

**새 파일:**
```
src/ml_platform/analysis/
├── __init__.py
├── models.py          # PaperAnalysis, FiveW1H, StrengthWeakness
├── analyzer.py        # PaperAnalyzer 클래스
├── prompts.py         # LLM 프롬프트 (Few-shot 포함)
├── reference_chain.py # 참고문헌 체인 추출기
└── evidence.py        # 근거 문장 추출 + Self-Correction
```

**태스크:**
1. `PaperAnalysis` Pydantic 모델 (5W1H + Strengths/Weaknesses/FutureWork + references + evidence_sentences)
2. LLM 프롬프트 — Few-shot 예제 포함, 논문 텍스트 → JSON 구조화
3. `PaperAnalyzer` — PDF/텍스트 → PaperAnalysis (Ollama API)
4. `ReferenceChainExtractor` — reference 목록 추출 + DOI/arXiv ID 정규화
5. `EvidenceExtractor` — 각 추출 항목의 근거 문장 함께 저장
6. Self-Correction 루프 — 1차 추출 결과를 LLM이 재검증
7. DB 스키마 확장 — `analysis_results` 테이블 + `entity_registry` (Entity Resolution)
8. CLI: `ml-research analyze <paper-id>`

**의존성:** 기존 httpx + pydantic + Ollama

### Phase 6: Knowledge Graph Builder (graphqlite)

**목표:** 분석된 논문들을 per-topic 독립 그래프로 구축

**Entity Resolution (Red Team 피드백 반영):**
- DOI, arXiv ID를 기반으로 논문 Unique Identifier 보장
- 동일 논문이 여러 소스에서 다른 이름으로 들어와도 하나의 노드로 통합
- `entity_registry` 테이블: `canonical_id → [doi, arxiv_id, semantic_scholar_id, title_variants]`

**노드 유형:**
| Label | 속성 | 설명 |
|-------|------|------|
| Paper | title, year, citation_count, composite_score, doi, arxiv_id | 논문 |
| Author | name, affiliation | 연구자 |
| Method | name, category | 방법론 |
| Dataset | name, domain | 데이터셋 |
| Task | name, domain | 해결 과제 |

**엣지 유형:**
| Rel Type | 설명 | 검증 |
|----------|------|------|
| CITES | A가 B를 인용 | reference_chain 기반 (deterministic) |
| IMPROVES | A가 B를 개선 | LLM 추출 + Evidence Sentence |
| EXTENDS | A가 B를 확장 | LLM 추출 + Evidence Sentence |
| CONTRADICTS | A가 B에 반박 | LLM 추출 + Evidence Sentence |
| USES_METHOD | 논문이 방법론 사용 | How 섹션에서 추출 |
| USES_DATASET | 논문이 데이터셋 사용 | How 섹션에서 추출 |
| SOLVES_TASK | 논문이 과제 해결 | What 섹션에서 추출 |
| AUTHORED_BY | 논문-저자 관계 | 메타데이터 기반 |

**Graph Validation Step (Red Team 피드백 반영):**
- LLM이 추출한 IMPROVES/EXTENDS/CONTRADICTS 엣지를 샘플링 검증
- Evidence Sentence가 실제 논문 텍스트에 존재하는지 확인
- 신뢰도 점수(confidence)를 엣지 속성으로 저장

**새 파일:**
```
src/ml_platform/graph/
├── __init__.py
├── models.py             # NodeType, EdgeType enum
├── entity_resolver.py    # DOI/arXiv ID 기반 Entity Resolution (NEW)
├── graph_builder.py      # TopicGraphBuilder
├── relation_extractor.py # LLM 관계 추출 + Few-shot
├── validator.py          # Graph Validation Step (NEW)
├── query.py              # Cypher 쿼리 헬퍼
├── merge.py              # 그래프 병합 (선택)
└── visualizer.py         # pyvis HTML (중요도 필터링 포함)
```

**태스크:**
1. NodeType/EdgeType enum + confidence 속성 정의
2. `EntityResolver` — DOI/arXiv ID 정규화 → canonical_id 매핑
3. `TopicGraphBuilder` — PaperAnalysis → graphqlite 그래프
4. `RelationExtractor` — Few-shot + LLM 관계 분류
5. `GraphValidator` — 샘플링 검증, evidence 존재 확인
6. Cypher 쿼리 헬퍼 (영향력 논문, 개선 체인, 커뮤니티)
7. `GraphVisualizer` — pyvis + 중요도 필터링 (100개+ 노드 대응)
8. `GraphMerger` — 독립 그래프 병합
9. CLI: `ml-research graph build/visualize/merge <topic>`

**의존성:** `graphqlite>=0.4.4`, `pyvis>=0.3`

### Phase 7: HTMX Web Dashboard

**목표:** 모든 기능을 웹에서 접근 가능한 풀 대시보드

**기술 구성:**
```
FastAPI (서버)
  ├── Jinja2 템플릿 (HTML 렌더링)
  ├── HTMX (동적 인터랙션, JS 없이)
  ├── Tailwind CSS CDN (스타일링, 빌드 없이)
  └── SSE (Server-Sent Events, DeepCode 진행 상태)
```

**페이지 구성:**

| Route | 기능 | HTMX 인터랙션 |
|-------|------|---------------|
| `/` | 홈 — 주제 검색 | 검색 → 결과 리스트 교체 |
| `/research/<topic>` | 연구 대시보드 | 논문 리스트 + 필터 + 정렬 |
| `/paper/<id>` | 논문 상세 | 5W1H 탭, 장단점, 관계 링크 |
| `/paper/<id>/implement` | DeepCode 구현 요청 | SSE로 진행 상태 스트리밍 |
| `/graph/<topic>` | 그래프 시각화 | pyvis iframe, 필터 컨트롤 |
| `/trends/<topic>` | 트렌드 리포트 | 동적 차트, Gap 리스트 |
| `/implementations` | 구현 이력 | 상태 필터, 로그 뷰 |

**새 파일:**
```
src/ml_platform/web/

├── routes/
```

6. 그래프 페이지 — pyvis iframe + HTMX 필터 컨트롤
7. 트렌드 리포트 페이지
8. **DeepCode 구현 요청** — "Implement" 버튼 → SSE 진행 상태

### Phase 8: Research Trend Analyzer

**목표:** 그래프 기반 트렌드 분석, 장단점 비교, 연구 Gap 식별

**분석 기능:**
1. **트렌드 감지** — 연도별 방법론/키워드 인기도 변화
2. **방법론 비교** — 동일 Task 내 Strengths/Weaknesses 매트릭스
3. **개선 체인 추적** — IMPROVES/EXTENDS 엣지 순회
4. **연구 Gap 식별** — Future Work 중 후속 연구 없는 영역
5. **핵심 논문 식별** — PageRank
6. **미래 방향 예측** — LLM 종합 리포트

**새 파일:**
```
src/ml_platform/trends/
├── __init__.py
├── analyzer.py      # TrendAnalyzer
├── comparator.py    # MethodComparator
├── gap_detector.py  # GapDetector
└── report.py        # TrendReportGenerator
```

**태스크:**
1. `TrendAnalyzer` — 연도별 노드/엣지 분포, 키워드 빈도
2. `MethodComparator` — 장단점 매트릭스
3. `GapDetector` — 해결되지 않은 Future Work
4. `TrendReportGenerator` — LLM 종합 리포트
5. 웹 라우트 연결 — `/trends/<topic>`
6. CLI: `ml-research trend-report <topic>`

### Phase 9: DeepCode Integration (Web Triggered)

**목표:** 웹에서 사용자가 승인한 논문만 DeepCode로 구현

**핵심 플로우:**
```
웹 논문 상세페이지 → [Implement with DeepCode] 버튼
  → 사용자가 구현 범위 선택 (전체 / 특정 메서드 / 특정 모델)
  → 백그라운드 DeepCode 실행
  → SSE로 진행 상태 스트리밍 (웹)
  → 완료 → 결과 코드 다운로드 / GitHub Push
```

**태스크:**
1. 기존 `deepcode_runner.py` 리팩토링 — 웹 서비스 레이어 분리
2. `ImplementationRequest` 모델 — 논문, 범위, 옵션
3. 백그라운드 태스크 큐 — FastAPI BackgroundTasks 또는 asyncio.Queue
4. SSE 엔드포인트 — 진행 상태 스트리밍
5. 구현 이력 DB — `implementations` 테이블
6. 결과 페이지 — 코드 뷰어, 다운로드, GitHub Push 버튼
